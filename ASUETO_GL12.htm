12.1. Общие сведения

Первая редакция стандарта МЭК 61850 «Communication networks and systems in substations» («Сети и системы связи на подстанциях») была разработана для систем связи подстанции. Стандарт определял различные аспекты цифрового обмена в системах автоматизации подстанции в следующих основных разделах [7,8].

МЭК 61850-1: Введение и общий обзор.

МЭК 61850-2: Глоссарий терминов.

МЭК 61850-3: Основные требования.

МЭК 61850-4: Управление системой и проектированием.

МЭК 61850-5: Требования связи к функциям и моделям устройств.

МЭК 61850-6: Язык описания конфигурации связи между микропроцессор­ными электронными устройствами подстанций.

МЭК 61850-7: Основная структура связи для оборудования подстанции и пита­ющей линии (4 части).

МЭК 61850-8-1: Описание специфиче­ского сервиса связи (SCSM) ─ Описание передачи данных по протоколу MMS (ИСО/МЭК 9506 - Часть 1 и Часть 2) и по протоколу ИСО/МЭК 8802-3.

МЭК 61850-9-1: Описание специфического сервиса связи (SCSM) ─  Выбороч­ные значения по последовательному ненаправленному многоточечному каналу передачи данных типа точка-точка.

МЭК 61850-9-2: Описание специфиче­ского сервиса связи (SCSM) ─  Выборочные значения по ИСО/МЭК 8802-3.

В настоящее время действует вторая редакция стандарта, которая включает более широкий круг электроустановок «Communication networks and systems for power utility automation» (в частности гидроэлектростанции и ветроэлектростанции).

Как уже отмечалось, основной целью стандарта является обеспечение совместимости работы в рамках АСУ ТП подстанции ИЭУ различных производителей. Для достижения этой цели необходимо:

а)   устройства должны подключаться на общую шину с общим протоколом;

б)  устройства должны понимать информацию, предоставленную другими устройствами;

в)  если применимо, устройства должны вместе выполнять общие или связанные функции (распределенные функции).

Для реализации п. а) в качестве общей шины в МЭК 61850 была выбрана сеть Ethternet, которая обеспечивает высокую скорость передачи данных (100 Мбит/с) и возможность общения устройств на одном уровне.

Для реализации п. б) и в) необходимо иметь единый базовый набор терминов и связывающую их информационную модель. Для разработки такой модели был произведен анализ всего первичного оборудования подстанций, всех функций, которые должны выполнять при автоматизации, и данных, которыми эти функции должны обмениваться.

 

 

12.2. Логические узлы и данные

Минимальная часть функции, выполняющая определенные действия, которая обменивается данными, называется логическим узлом (LN).

Каждое ИЭУ представляет собой физическое устройство, которое подключено к сети и имеет свой сетевой адрес. В каждом физическом устройстве может быть одно или несколько логических устройств. Каждое логическое устройство может содержать несколько логических узлов. Модель логического устройства по стандарту МЭК 61850 позволяет одному физическому устройству функционировать в качестве модуля-посредника или машины-шлюза для многих устройств.

Выделены стандартные типы логических узлов (в скобках указано их количество в стандарте):

– L– Системные (2);

– R – Относящиеся к РЗ (10);

– C –  Control – Управление (5);

– G – Generic – Общие (3);

– I – Interfacing/archiving – Интерфейс/архив (4);

– A – Automatic control – Автоматическое управление (4);

– M – Metering/measurement – Измерения (8);

– S – Sensor/monitoring – Датчики (4);

– X – Switchgear – КА (2);

– T Instrument transformers – ТТ/ТН (2);

– Y – Силовые трансформаторы (4);

– Z – Зарезервировано на будущее (15);

Внутри каждого типа введены стандартные обозначения узлов, состоящие из четырех символов, например:

– PDIF– дифференциальная защита;

– RBRF – устройство резервирования отказов выключателя;

–XCBR – выключатель;

– CSWI – управление коммутационными аппаратами;

– MMXU – измерительное устройство;

–YPTR– силовой трансформатор.

Каждый логический узел содержит один или несколько элементов данных. Каждый элемент данных имеет собственное имя. Имена определяются Стандартом и отображают функции энергосистемы. Например, выключатель смоделирован как логический узел XCBR. Имя узла содержит ряд данных, в том числе обозначения:

• Loc – для определения режима управления («дистанционный» или «локальный»);

• OpCnt – для подсчета операций;

• Pos – для определения положения;

• BlkOpn – для блокировки команд отключения выключателя;

• BlkCls – для блокировки команд включения выключателя;

• CBOpCap – для возможности срабатывания выключателя.

Поскольку объекты данных состоят из общих стандартных частей (таких как Состояние, Управление, Измерение, Замена), была разработана концепция классов общих данных (или CDC), которые определяют стандартные составные элементы, с помощью которых можно создать более сложные составные объекты данных. Данные могут быть обязательными или дополнительными (Mandatory/Optional).

 

Рис. 12.1. Структура логического узла выключателя XCBR

 

Каждый элемент данных в логическом узле соответствует техническим характеристикам класса общих данных (CDC) согласно проколу МЭК 61850-7-3. Каждый класс общих данных описывает тип и структуру данных в логическом узле. Например, существуют классы CDC для данных состояния, измеряемых данных, данных регулируемого состояния, данных регулируемых аналоговых уставок, уставок состояния и аналоговых уставок. Каждый класс общих данных имеет определенное имя и ряд свойств класса – каждое свойство с определенным именем, определенным видом и конкретной целью. Каждое индивидуальное свойство класса CDC принадлежит определенной категории, которая образована по признаку функциональных ограничений (FC). Например, в классе Состояния одной точки (SPS), представленном на рис.12.2, введены функциональные ограничения для: свойств состояния – ST, свойств подстановочных значений – SV, свойств описания – DC, свойств расширенного определения – EX. В этом примере свойства состояния класса SPS состоят из stVal (значения состояния), q (признака качества) и t (метки времени).

12.3. Язык описания моделей

Для описания информационных моделей в МЭК 61850 используется стандартизированный язык описания конфигурации – Substation Configuration Language (SCL). Этот язык основывается на языке XML, получившем широкое распространение при моделировании данных. Описание данных на XML в чем-то похоже на язык HTML, используемый при описании интернет-страниц. Особенностью языка XML является то, что описание модели может быть прочитано и однозначно интерпретировано как компьютером, так и человеком. 

Основные виды файлов, в которых хранится описание модели данных на языке SCL:

*.SSD – содержит описание топологии подстанции (однолинейную схему) и требуемой функциональности АСУ ТП (логические узлы без привязки к конкретным ИЭУ);

*.ICD – содержит описание функциональных возможностей ИЭУ (для каждого устройства – свой файл);

*.SСD – описывает подстанцию целиком, включая топологию, ИЭУ, функциональность и связи.

*.CID – содержит описание функционала ИЭУ, в конкретном проекте, включая привязку к сетевым адресам и информационный обмен между устройствами (для каждого устройства – свой файл);

Хотя инструменты конфигурации (программного обеспечения для автоматизации проектирования) различаются у различных производителей, процедура создания модели данных проекта подстанции практически аналогична (рис. 12.3). Во-первых, требуется создать проект подстанции или как пустой шаблон, или с какой-либо системной информацией, импортируя файл описания подстанции (SSD). Затем к описанию подстанции добавляются описания ИЭУ. Поскольку каждое ИЭУ представлено соответствующим ICD-файлом, ICD-файлы импортируются в проект подстанции, а системный конфигуратор проверяет достоверность ICD-файлов во время процесса импортирования. Если ICD-файлы успешно импортированы в проект подстанции, возможно, потребуется дополнительно выполнить действия, чтобы добавить ИЭУ к подстанции.

Как только все ИЭУ вводятся в подстанцию, можно продолжить конфигурацию следующим образом:

• назначить индивидуальные адреса сети ИЭУ;

• назначить в соответствии с требованиями заказчика префиксы логических узлов

• создать перекрестную связь (конфигурируя GOOSE-сообщения для их передачи от одного ИЭУ к другим).

По завершении системных конфигураций, результаты сохраняются в SCD-файле, в котором содержится не только конфигурация для каждого ИЭУ на подстанции, но также и системная конфигурация для всей подстанции. На основе SCD-файла формируются CID-файлы для обновления конфигурации ИЭУ.

SCD-файл, как минимум, состоит из пяти основных взаимосвязанных секций:

• заголовок (общее описание проекта);

• подстанция (описание первичного оборудования и связанных с ним логических узлов);

• связь (структура сети Ethernet с присвоением IP-адресов всем устройствам);

• ИЭУ-секция (для каждого устройства с описанием физических и логических устройств, а также выполняемых на них логических узлов);

• шаблоны типа данных (описание используемых классов данных).

12.4. Цифровой обмен данными

Абстрактные данные и модели объектов по стандарту МЭК 61850 определяют стандартизированный метод описания устройств энергосистемы, который позволяет всем микропроцессорным электронным устройствам представлять данные, используя структуры, идентичные соответствующим функциям энергосистемы. Физически передача данных осуществляется по сети Ethernet. МЭК 61850 описывает преобразование абстрактных объектов и сервисов в протоколы сети. В стандарте рассматриваются два вида цифровых шин:

– шина подстанции, по которой ведется обмен данными между АРМ, серверами и ИЭУ;

– шина процесса, по которой информация от полевых устройств (датчиков и исполнительных механизмов) поступает в ИЭУ не по контрольному («медному») кабелю, а в оцифрованном виде.

Можно выделить три основные группы данных.

1.       Некритичные ко времени передачи, в частности, обмен между АРМ оперативного персонала и ИЭУ, где время определяется реакцией оператора.

2.  Критичные ко времени передачи, в частности, информация о состоянии коммутационных аппаратов, передаваемая непосредственно между ИЭУ и участвующая в алгоритмах управления. Время передачи порядка нескольких миллисекунд (4 мс для команд отключения).

3.  Цифровая информация от трансформаторов тока и напряжения SV (Sampled Value). Эти данные передаются по шине процесса. Нагрузка на сеть весьма велика, т.к. необходимо передавать значения токов и напряжений с дискретностью 80 точек на период (20 мс) для задач РЗА или 250 точек для коммерческого учета.

В первом случае применяется технология клиент-сервер. АРМ выступает в качестве клиента, а ИЭУ в качестве сервера. Передача информации осуществляется стандартными средствами TCP/IP, применяемыми, в частности в Internet. Для передачи данных в соответствии с моделью данных используется стандартный протокол сети Ethernet MMS (Manufacturing Message Specification). От сетевых коммутаторов не требуется каких-то «особых способностей».

Во втором случае применяется технология подписчик – издатель. Информация, выданная любым устройством («издателем») может быть использована любыми другими устройствам («подписчиками»), одним или несколькими. Пример описания сообщений между устройствами управления линией AA1E1Q01A1 и трансформатором AA1E1Q02A1 для организации оперативной блокировки разъединителей рассмотрен в предыдущем параграфе. Для передачи данных используется протокол сети GOOSE (Generic Object Oriented Substation Event). GOOSE-сообщения имеют более высокий приоритет, чем стандартные пакеты Ethernet. Поэтому к сетевым коммутаторам предъявляется требование организации ускоренной доставки высокоприоритетных сообщений (рис. 12.6).

Для повышения надежности доставки GOOSE-сообщений, если переменные в сообщении не меняли состояния, то передача ведется с заданной частотой; если переменная изменилась (например, коммутационный аппарат включился или отключился), то передача сообщения переходит на повышенную частоту с постепенным возвратом на заданную.
